\doxysubsection{Detailed Description}
\+\_\+\+\_\+\+\_\+\+MANBRIEF\+\_\+\+\_\+\+\_\+ Stream memory operations of the low-\/level CUDA driver API (\+\_\+\+\_\+\+\_\+\+CURRENT\+\_\+\+FILE\+\_\+\+\_\+\+\_\+) \+\_\+\+\_\+\+\_\+\+ENDMANBRIEF\+\_\+\+\_\+\+\_\+

This section describes the stream memory operations of the low-\/level CUDA driver application programming interface.

The whole set of operations is disabled by default. Users are required to explicitly enable them, e.\+g. on Linux by passing the kernel module parameter shown below\+: modprobe nvidia NVreg\+\_\+\+Enable\+Stream\+Mem\+OPs=1 There is currently no way to enable these operations on other operating systems.

Users can programmatically query whether the device supports these operations with \doxylink{group___c_u_d_a___d_e_v_i_c_e_ga5a73f28b53203cd6097fc48d97ca4ead}{cu\+Device\+Get\+Attribute()} and \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa3fcdc07d0e356a84b6c02730df6d43db}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OPS}.

Support for the \doxylink{group___c_u_d_a___t_y_p_e_s_gga5ba489253eb905a983228a810c1f3360a2f10c357c1882ae7b0b947225c1a0d55}{CU\+\_\+\+STREAM\+\_\+\+WAIT\+\_\+\+VALUE\+\_\+\+NOR} flag can be queried with \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aabe06f9c31c264e0b29be6cc6330b5047}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+\+STREAM\+\_\+\+WAIT\+\_\+\+VALUE\+\_\+\+NOR}.

Support for the \doxylink{group___c_u_d_a___m_e_m_o_p_gaddd21e0851bb4ee05946904dd3a1d55a}{cu\+Stream\+Write\+Value64()} and \doxylink{group___c_u_d_a___m_e_m_o_p_gaa70681aba6f1d83b169d7ce8ca97a657}{cu\+Stream\+Wait\+Value64()} functions, as well as for the \doxylink{group___c_u_d_a___t_y_p_e_s_ggad22b1c6c35ae110dd96d576d33c14f1fa7e4ff0402540ee028226c47858cf8a99}{CU\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OP\+\_\+\+WAIT\+\_\+\+VALUE\+\_\+64} and \doxylink{group___c_u_d_a___t_y_p_e_s_ggad22b1c6c35ae110dd96d576d33c14f1fa6e576dc8791fa4abf7a3b36e428cd8de}{CU\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OP\+\_\+\+WRITE\+\_\+\+VALUE\+\_\+64} flags, can be queried with \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa20db4c9e8d81eb05a937d4ed5c132575}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+64\+\_\+\+BIT\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OPS}.

Support for both \doxylink{group___c_u_d_a___t_y_p_e_s_gga5ba489253eb905a983228a810c1f3360a5dd93e7173619c943fae495568f4d771}{CU\+\_\+\+STREAM\+\_\+\+WAIT\+\_\+\+VALUE\+\_\+\+FLUSH} and \doxylink{group___c_u_d_a___t_y_p_e_s_ggad22b1c6c35ae110dd96d576d33c14f1fad4e126a89b57ebf4a213ee1774ed103f}{CU\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OP\+\_\+\+FLUSH\+\_\+\+REMOTE\+\_\+\+WRITES} requires dedicated platform hardware features and can be queried with \doxylink{group___c_u_d_a___d_e_v_i_c_e_ga5a73f28b53203cd6097fc48d97ca4ead}{cu\+Device\+Get\+Attribute()} and \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa4c702e528f71bf513ee0f3f32878ce9d}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+FLUSH\+\_\+\+REMOTE\+\_\+\+WRITES}.

Note that all memory pointers passed as parameters to these operations are device pointers. Where necessary a device pointer should be obtained, for example with cu\+Mem\+Host\+Get\+Device\+Pointer().

None of the operations accepts pointers to managed memory buffers (\doxylink{group___c_u_d_a___m_e_m_gab3c8db1c575451d53313b05b1c6db7f8}{cu\+Mem\+Alloc\+Managed}).

Batch operations to synchronize the stream via memory operations. 

This is a batch version of \doxylink{group___c_u_d_a___m_e_m_o_p_ga5653cac96aef1ea78757d264e75c3589}{cu\+Stream\+Wait\+Value32()} and \doxylink{group___c_u_d_a___m_e_m_o_p_ga3f0dab1d1fe1f885ae4128db7eddbb26}{cu\+Stream\+Write\+Value32()}. Batching operations may avoid some performance overhead in both the API call and the device execution versus adding them to the stream in separate API calls. The operations are enqueued in the order they appear in the array.

See \doxylink{group___c_u_d_a___t_y_p_e_s_ga92cf7b4a7fd965941030ccd6339e6dd6}{CUstream\+Batch\+Mem\+Op\+Type} for the full set of supported operations, and \doxylink{group___c_u_d_a___m_e_m_o_p_ga5653cac96aef1ea78757d264e75c3589}{cu\+Stream\+Wait\+Value32()}, \doxylink{group___c_u_d_a___m_e_m_o_p_gaa70681aba6f1d83b169d7ce8ca97a657}{cu\+Stream\+Wait\+Value64()}, \doxylink{group___c_u_d_a___m_e_m_o_p_ga3f0dab1d1fe1f885ae4128db7eddbb26}{cu\+Stream\+Write\+Value32()}, and \doxylink{group___c_u_d_a___m_e_m_o_p_gaddd21e0851bb4ee05946904dd3a1d55a}{cu\+Stream\+Write\+Value64()} for details of specific operations.

Basic support for this can be queried with \doxylink{group___c_u_d_a___d_e_v_i_c_e_ga5a73f28b53203cd6097fc48d97ca4ead}{cu\+Device\+Get\+Attribute()} and \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa3fcdc07d0e356a84b6c02730df6d43db}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OPS}. Wait on a memory location. 

Enqueues a synchronization of the stream on the given memory location. Work ordered after the operation will block until the given condition on the memory is satisfied. By default, the condition is to wait for (int32\+\_\+t)(\texorpdfstring{$\ast$}{*}addr -\/ value) \texorpdfstring{$>$}{>}= 0, a cyclic greater-\/or-\/equal. Other condition types can be specified via {\ttfamily flags}.

If the memory was registered via cu\+Mem\+Host\+Register(), the device pointer should be obtained with cu\+Mem\+Host\+Get\+Device\+Pointer(). This function cannot be used with managed memory (\doxylink{group___c_u_d_a___m_e_m_gab3c8db1c575451d53313b05b1c6db7f8}{cu\+Mem\+Alloc\+Managed}).

Support for this can be queried with \doxylink{group___c_u_d_a___d_e_v_i_c_e_ga5a73f28b53203cd6097fc48d97ca4ead}{cu\+Device\+Get\+Attribute()} and \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa3fcdc07d0e356a84b6c02730df6d43db}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OPS}.

Support for CU\+\_\+\+STREAM\+\_\+\+WAIT\+\_\+\+VALUE\+\_\+\+NOR can be queried with \doxylink{group___c_u_d_a___d_e_v_i_c_e_ga5a73f28b53203cd6097fc48d97ca4ead}{cu\+Device\+Get\+Attribute()} and \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aabe06f9c31c264e0b29be6cc6330b5047}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+\+STREAM\+\_\+\+WAIT\+\_\+\+VALUE\+\_\+\+NOR}.

Wait on a memory location. 

Enqueues a synchronization of the stream on the given memory location. Work ordered after the operation will block until the given condition on the memory is satisfied. By default, the condition is to wait for (int64\+\_\+t)(\texorpdfstring{$\ast$}{*}addr -\/ value) \texorpdfstring{$>$}{>}= 0, a cyclic greater-\/or-\/equal. Other condition types can be specified via {\ttfamily flags}.

If the memory was registered via cu\+Mem\+Host\+Register(), the device pointer should be obtained with cu\+Mem\+Host\+Get\+Device\+Pointer().

Support for this can be queried with \doxylink{group___c_u_d_a___d_e_v_i_c_e_ga5a73f28b53203cd6097fc48d97ca4ead}{cu\+Device\+Get\+Attribute()} and \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa20db4c9e8d81eb05a937d4ed5c132575}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+CAN\+\_\+\+USE\+\_\+64\+\_\+\+BIT\+\_\+\+STREAM\+\_\+\+MEM\+\_\+\+OPS}.
