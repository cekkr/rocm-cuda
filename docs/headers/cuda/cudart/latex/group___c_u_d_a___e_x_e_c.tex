Sets the shared memory configuration for a device function. 

On devices with configurable shared memory banks, this function will force all subsequent launches of the specified device function to have the given shared memory bank size configuration. On any given launch of the function, the shared memory configuration of the device will be temporarily changed if needed to suit the function\textquotesingle{}s preferred configuration. Changes in shared memory configuration between subsequent launches of functions, may introduce a device side synchronization point.

Any per-\/function setting of shared memory bank size set via \doxylink{group___c_u_d_a___e_x_e_c_gadee0dbc8805fa99d345d42a3077bac9a}{cu\+Func\+Set\+Shared\+Mem\+Config} will override the context wide setting set with \doxylink{group___c_u_d_a___c_t_x_gab94b06157868dc684d2d9faf48468d30}{cu\+Ctx\+Set\+Shared\+Mem\+Config}.

Changing the shared memory bank size will not increase shared memory usage or affect occupancy of kernels, but may have major effects on performance. Larger bank sizes will allow for greater potential bandwidth to shared memory, but will change what kinds of accesses to shared memory will result in bank conflicts.

This function will do nothing on devices with fixed shared memory bank size.

Launches a CUDA function where thread blocks can cooperate and synchronize as they execute. 

Invokes the kernel {\ttfamily f} on a {\ttfamily grid\+DimX} x {\ttfamily grid\+DimY} x {\ttfamily grid\+DimZ} grid of blocks. Each block contains {\ttfamily block\+DimX} x {\ttfamily block\+DimY} x {\ttfamily block\+DimZ} threads.

{\ttfamily shared\+Mem\+Bytes} sets the amount of dynamic shared memory that will be available to each thread block.

The device on which this kernel is invoked must have a non-\/zero value for the device attribute \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aae9d3b7631f9112048c541cdb08c8a4e6}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+COOPERATIVE\+\_\+\+LAUNCH}.

The total number of blocks launched cannot exceed the maximum number of blocks per multiprocessor as returned by \doxylink{group___c_u_d_a___o_c_c_u_p_a_n_c_y_gae81e36f2257cd84ef167e39ac98a4bc8}{cu\+Occupancy\+Max\+Active\+Blocks\+Per\+Multiprocessor} (or \doxylink{group___c_u_d_a___o_c_c_u_p_a_n_c_y_ga30ea39b7e266a235bfa08b593b1063b7}{cu\+Occupancy\+Max\+Active\+Blocks\+Per\+Multiprocessor\+With\+Flags}) times the number of multiprocessors as specified by the device attribute \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aab92d0a38a94a3f61d4c53e00114afcaa}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+MULTIPROCESSOR\+\_\+\+COUNT}.

The kernel cannot make use of CUDA dynamic parallelism.

Kernel parameters must be specified via {\ttfamily kernel\+Params}. If {\ttfamily f} has N parameters, then {\ttfamily kernel\+Params} needs to be an array of N pointers. Each of {\ttfamily kernel\+Params}\mbox{[}0\mbox{]} through {\ttfamily kernel\+Params}\mbox{[}N-\/1\mbox{]} must point to a region of memory from which the actual kernel parameter will be copied. The number of kernel parameters and their offsets and sizes do not need to be specified as that information is retrieved directly from the kernel\textquotesingle{}s image.

Calling \doxylink{group___c_u_d_a___e_x_e_c_gaeb5259b5a28ee4379d905bee4827a767}{cu\+Launch\+Cooperative\+Kernel()} sets persistent function state that is the same as function state set through \doxylink{group___c_u_d_a___e_x_e_c_ga920a0845a7a1e6dca2af5e51b9a00ab6}{cu\+Launch\+Kernel} API

When the kernel {\ttfamily f} is launched via \doxylink{group___c_u_d_a___e_x_e_c_gaeb5259b5a28ee4379d905bee4827a767}{cu\+Launch\+Cooperative\+Kernel()}, the previous block shape, shared size and parameter info associated with {\ttfamily f} is overwritten.

Note that to use \doxylink{group___c_u_d_a___e_x_e_c_gaeb5259b5a28ee4379d905bee4827a767}{cu\+Launch\+Cooperative\+Kernel()}, the kernel {\ttfamily f} must either have been compiled with toolchain version 3.\+2 or later so that it will contain kernel parameter information, or have no kernel parameters. If either of these conditions is not met, then \doxylink{group___c_u_d_a___e_x_e_c_gaeb5259b5a28ee4379d905bee4827a767}{cu\+Launch\+Cooperative\+Kernel()} will return \doxylink{group___c_u_d_a___t_y_p_e_s_gga0cdead942fd5028d157641eef6bdeeaaa0b7bd1dd2fb3491c588ce569c02d1a2f}{CUDA\+\_\+\+ERROR\+\_\+\+INVALID\+\_\+\+IMAGE}.

Launches CUDA functions on multiple devices where thread blocks can cooperate and synchronize as they execute. 

\begin{DoxyRefDesc}{Deprecated}
\item[\mbox{\hyperlink{deprecated__deprecated000005}{Deprecated}}]This function is deprecated as of CUDA 11.\+3.\end{DoxyRefDesc}


Invokes kernels as specified in the {\ttfamily launch\+Params\+List} array where each element of the array specifies all the parameters required to perform a single kernel launch. These kernels can cooperate and synchronize as they execute. The size of the array is specified by {\ttfamily num\+Devices}.

No two kernels can be launched on the same device. All the devices targeted by this multi-\/device launch must be identical. All devices must have a non-\/zero value for the device attribute \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aa1d10266c0f67ba1fe57b23c9d311f0a3}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+COOPERATIVE\+\_\+\+MULTI\+\_\+\+DEVICE\+\_\+\+LAUNCH}.

All kernels launched must be identical with respect to the compiled code. Note that any {\bfseries{device}}, {\bfseries{constant}} or {\bfseries{managed}} variables present in the module that owns the kernel launched on each device, are independently instantiated on every device. It is the application\textquotesingle{}s responsiblity to ensure these variables are initialized and used appropriately.

The size of the grids as specified in blocks, the size of the blocks themselves and the amount of shared memory used by each thread block must also match across all launched kernels.

The streams used to launch these kernels must have been created via either \doxylink{group___c_u_d_a___s_t_r_e_a_m_ga1546b970db0f91e162988701b1ac027e}{cu\+Stream\+Create} or \doxylink{group___c_u_d_a___s_t_r_e_a_m_gaea2fc94b2745094bf05cae5b1495de9a}{cu\+Stream\+Create\+With\+Priority}. The NULL stream or \doxylink{group___c_u_d_a___t_y_p_e_s_gaa53e8210837f039dd6434a3a4c3324aa}{CU\+\_\+\+STREAM\+\_\+\+LEGACY} or \doxylink{group___c_u_d_a___t_y_p_e_s_ga02e40b82600f62c42ed29abb150f857c}{CU\+\_\+\+STREAM\+\_\+\+PER\+\_\+\+THREAD} cannot be used.

The total number of blocks launched per kernel cannot exceed the maximum number of blocks per multiprocessor as returned by \doxylink{group___c_u_d_a___o_c_c_u_p_a_n_c_y_gae81e36f2257cd84ef167e39ac98a4bc8}{cu\+Occupancy\+Max\+Active\+Blocks\+Per\+Multiprocessor} (or \doxylink{group___c_u_d_a___o_c_c_u_p_a_n_c_y_ga30ea39b7e266a235bfa08b593b1063b7}{cu\+Occupancy\+Max\+Active\+Blocks\+Per\+Multiprocessor\+With\+Flags}) times the number of multiprocessors as specified by the device attribute \doxylink{group___c_u_d_a___t_y_p_e_s_gga3b9f561d2a42733dde99b2cedcaa413aab92d0a38a94a3f61d4c53e00114afcaa}{CU\+\_\+\+DEVICE\+\_\+\+ATTRIBUTE\+\_\+\+MULTIPROCESSOR\+\_\+\+COUNT}. Since the total number of blocks launched per device has to match across all devices, the maximum number of blocks that can be launched per device will be limited by the device with the least number of multiprocessors.

The kernels cannot make use of CUDA dynamic parallelism.

Enqueues a host function call in a stream. 

Enqueues a host function to run in a stream. The function will be called after currently enqueued work and will block work added after it.

The host function must not make any CUDA API calls. Attempting to use a CUDA API may result in \doxylink{group___c_u_d_a___t_y_p_e_s_gga0cdead942fd5028d157641eef6bdeeaaae23d0197c490ec332a43e55b167968a3}{CUDA\+\_\+\+ERROR\+\_\+\+NOT\+\_\+\+PERMITTED}, but this is not required. The host function must not perform any synchronization that may depend on outstanding CUDA work not mandated to run earlier. Host functions without a mandated order (such as in independent streams) execute in undefined order and may be serialized.

For the purposes of Unified Memory, execution makes a number of guarantees\+: 
\begin{DoxyItemize}
\item The stream is considered idle for the duration of the function\textquotesingle{}s execution. Thus, for example, the function may always use memory attached to the stream it was enqueued in. 
\item The start of execution of the function has the same effect as synchronizing an event recorded in the same stream immediately prior to the function. It thus synchronizes streams which have been "{}joined"{} prior to the function. 
\item Adding device work to any stream does not have the effect of making the stream active until all preceding host functions and stream callbacks have executed. Thus, for example, a function might use global attached memory even if work has been added to another stream, if the work has been ordered behind the function call with an event. 
\item Completion of the function does not cause a stream to become active except as described above. The stream will remain idle if no device work follows the function, and will remain idle across consecutive host functions or stream callbacks without device work in between. Thus, for example, stream synchronization can be done by signaling from a host function at the end of the stream. 
\end{DoxyItemize}

Note that, in contrast to \doxylink{group___c_u_d_a___s_t_r_e_a_m_ga120776d5935ae856369b231dc6dedcbf}{cu\+Stream\+Add\+Callback}, the function will not be called in the event of an error in the CUDA context.

Launches a CUDA function. 

Invokes the kernel {\ttfamily f} on a {\ttfamily grid\+DimX} x {\ttfamily grid\+DimY} x {\ttfamily grid\+DimZ} grid of blocks. Each block contains {\ttfamily block\+DimX} x {\ttfamily block\+DimY} x {\ttfamily block\+DimZ} threads.

{\ttfamily shared\+Mem\+Bytes} sets the amount of dynamic shared memory that will be available to each thread block.

Kernel parameters to {\ttfamily f} can be specified in one of two ways\+:

1) Kernel parameters can be specified via {\ttfamily kernel\+Params}. If {\ttfamily f} has N parameters, then {\ttfamily kernel\+Params} needs to be an array of N pointers. Each of {\ttfamily kernel\+Params}\mbox{[}0\mbox{]} through {\ttfamily kernel\+Params}\mbox{[}N-\/1\mbox{]} must point to a region of memory from which the actual kernel parameter will be copied. The number of kernel parameters and their offsets and sizes do not need to be specified as that information is retrieved directly from the kernel\textquotesingle{}s image.

2) Kernel parameters can also be packaged by the application into a single buffer that is passed in via the {\ttfamily extra} parameter. This places the burden on the application of knowing each kernel parameter\textquotesingle{}s size and alignment/padding within the buffer.
